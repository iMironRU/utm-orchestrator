# utm-orchestrator — беклог и архитектура

Живой документ. Составлен 2026-07-10 на основе всего, что проверено на боевой
машине (см. `NOTES.md` — там доказательства и механика). Реализуем поэтапно.

## Статус на 2026-07-11 (что сделано / где мы)

Локальный оркестратор в бою на живой машине (6 УТМ). Сделано и проверено вживую:
- **Boot-подъём УТМ** службой (introduce, session 0) — unattended, проверен ребутами.
- **Служба** (session 0) + **панель** (Kestrel, localhost:8090) + **трей**
  (single-instance, автозапуск Run-ключ + задача «при входе»).
- **Веб-панель на ЖИВЫХ данных** (демо убрано), честные «в разработке» заглушки.
- **Точечный перезапуск УТМ** (introduce), **смена порта УТМ**, **heal**
  (рестарт SCardSvr + подъём) — всё endpoints службы, без UAC.
- **Файрвол**: подсветка «открыт/закрыт» + управление (своё правило на УТМ).
- Версия УТМ в карточках; кликабельные плитки/порт.
- Консолидация в `C:\UtmOrchestrator` (data/logs/wwwroot/exports).
- Первый запуск: обследование + **подхват** существующих УТМ (adopt).
- Флаг **bringUp → «Запускается…»** в трее/вебе.
- **CI/CD**: push в main → сборка на Windows → релиз `latest` (архив + Setup.exe).
- **Инсталлятор** `UtmOrchestrator-Setup.exe` (скачал→кликнул): качает пейлоад →
  `install.ps1` → открывает панель. [E2E-тест на чистой машине — за оператором]
- **Перенос УТМ — ЭКСПОРТ** в бандл (софт+база+служба+манифест+procrun) готов и проверен.

Ждёт второй/чистой машины (ПАУЗА):
- **Перенос УТМ — ИМПОРТ** (приёмник): пересоздание procrun-службы + ре-поиск
  ридера по ФСРАР (имя ридера машинно-зависимо); проверяется только на 2-м компе с токеном.
- E2E-тест инсталлятора (двойной клик на чистой машине).

На паузе по решению оператора:
- **Централизованное управление парком** (после локального).

Связка веб↔трей: реализована; **scan** токенов идёт через трей (интерактивная
PKCS11-сессия), **heal** переехал в службу (UAC не нужен).

## 0. Главные принципы (не нарушать)

- **Не ломать работающее.** Любая операция с ридерами меняет ГЛОБАЛЬНУЮ
  PC/SC-таблицу и может сломать другие УТМ. Здоровые УТМ не трогаем никогда;
  чиним только сбойные и только точечно.
- **Идемпотентность и безопасный повтор.** Любой шаг можно прервать/повторить
  без порчи состояния. Перед действием — читаем реальное состояние, потом
  решаем.
- **Отказоустойчивость по частям.** Если один/несколько УТМ не поднимаются —
  остальные работают. Сбойные помечаем: статус + причина (нет токена, нет
  папки, порт занят, БД повреждена, и т.д.).
- **Серийник токена — первичный ключ** (PKCS11 `C_GetTokenInfo.serialNumber`,
  hex). ФСРАР-ИД — вторичный (может отсутствовать у пустого токена, появляется
  после выпуска подписи). Device-имя ридера НЕ использовать для идентификации
  (самореференциально, хрупко).
- **Всё логируется**, пошагово, по каждому УТМ, с ротацией.
- **Разработка на боевой машине — только безопасно.** Пишущие/деструктивные
  операции тестируем в контролируемом окне или на отдельных портах/папках; по
  умолчанию — read-only.

## 1. Поддерживаемые состояния машины (при установке/подхвате)

- **Чистая**: нет УТМ, нет 2UTM, нет нас.
- **Есть один УТМ** (вендорская установка) — подхватить.
- **Есть 2UTM** — сосуществовать/мигрировать (снять его автозапуск, перенять
  управление; НЕ ломать, пока не подтверждено).
- **Есть мы** (наша прошлая установка) — обновить/переконфигурировать.

## 2. Поддерживаемые состояния токенов

- Один токен / много токенов.
- «Чистый» токен (без сертификатов).
- Токен без ФСРАР-ИД, но с одним/несколькими другими сертификатами.
- Токен с ФСРАР-ИД (готовый).
- Токен физически «уснул» (ридер есть, карта не отвечает) → лечится
  программным сбросом (forget + SCardSvr restart) или переподключением.

## 3. Ключевая механика (проверено, см. NOTES)

- **RSA-путь при ЗАГРУЗКЕ читает строго слот 0.** В рантайме переиниц. НЕ
  читает слот 0 (держит сессию/ищет по идентичности). Значит: каждый УТМ должен
  стартовать, когда ЕГО токен на слоте 0 → службы поднимаем ПО ОДНОЙ на каждой
  загрузке ОС (не Windows auto-start скопом).
- **Peel-down bring-up (надёжный, без хрупких rename):**
  1. Программный сброс → детерминированный нативный порядок.
  2. Читаем FSRAR/serial на слоте 0 → находим его службу → запускаем → ждём
     готовности (служба «схватывает» токен).
  3. `SCardForgetReader` ридера слота 0 → на слот 0 встаёт следующий токен.
  4. Повтор, пока не поднимутся все.
  5. Программный сброс (вернуть все ридеры для рантайма).
- **Установка «с нуля» без вендорского setup.exe**: копируем распакованный
  источник (innoextract) → папки, правим порты + PIN, procrun-служба,
  choreography. Проверено на 6 токенах.
- **Внутренние порты** (схема 2UTM): web = задан; transport = 8192 − i;
  agent = 8194 + i (i — номер экземпляра; UTM_1/base = 8080/8192/8194).

## 4. Компоненты

- **core** — библиотека: сканер токенов (serial+fsrar+slot+reader), ReaderTable
  (winscard: list/attr/forget/introduce + программный reset), ServiceControl
  (procrun install/delete, sc start/stop, ожидание готовности по HTTP),
  HTTP-диагностика (`/api/info/list` → ownerId/rsaError), лог.
- **state** — единый файл состояния (serial→port→service→fsrar→version, ссылка
  на PIN, статусы/причины сбоев). + блокировка параллельного доступа.
- **orchestrator-service** — Windows-служба (сессия 0): boot-bring-up,
  периодический health-check + точечное восстановление, слежение за версией
  УТМ и за релизами нашего оркестратора на GitHub. UI не рисует (изоляция
  сессий).
- **tray** — маленькое приложение в сессии пользователя (автозапуск при входе):
  иконка в трее со сводным статусом (зелёная/красная+число/анимация/нет связи),
  меню быстрых действий (открыть панель, поднять все, перепроверить, логи),
  общается со службой по локальному API. Служба сама трей показать не может —
  отсюда отдельный компонент.
- **panel** — веб-интерфейс (локально + удалённо/телефон), аутентификация,
  мониторинг, действия (install/update/remove/add/attach, порты, firewall,
  логи). Дизайн — см. `DESIGN_BRIEF.md`.
- **installer/updater** — установка/подхват/удаление УТМ, обновление УТМ с
  сохранением transportDB/conf/l (логика Update-UTM.ps1), самообновление
  оркестратора.

## Идея: связка веб ↔ трей через очередь заданий (для интерактивных операций)

Проблема: часть операций требует ИНТЕРАКТИВНОЙ сессии, служба (session 0) их не может:
- PKCS11-скан физического токена (для мастера установки нового УТМ);
- рестарт SCardSvr («Полечить токены» — разбудить замёрзший токен).

Решение: когда панель открыта с localhost, оператор за машиной и трей запущен в его
сессии. Делаем очередь заданий через службу:
```
Веб ──POST /api/jobs {type:scan|heal|...}──▶ Служба ведёт очередь (файл в %ProgramData%)
Трей (интерактивная сессия) ── поллит GET /api/jobs/pending раз в 1-2с ──▶ берёт job
   ── выполняет в своей сессии (PKCS11-скан / heal) ── POST /api/jobs/{id}/result ──▶
Веб ── поллит GET /api/jobs/{id} ── показывает результат
```
- Служба — единая точка API + очередь. Трей — «руки» веба для интерактивного.
- Если трея нет (удалённо/без входа) — задания типа scan/heal недоступны (гасим кнопки).
  Это логично: установка/лечение и так требуют оператора у машины.
- Даёт: мастер установки В ВЕБЕ (скан токенов делегируется трею) + «Полечить токены»
  из веба. Split чистый: служба = session-0-безопасное, трей = интерактивное, веб = UI.
Статус: спроектировано, к реализации (после текущей полировки веба).

## Первый запуск: обследование машины + выбор действия
При первом старте (или по кнопке «Обследовать») оркестратор сканирует машину:
- установленные службы Transport* (UtmDiscovery уже умеет: папка, порт, ФСРАР);
- подключённые токены (серийники/ФСРАР — интерактивно через трей);
- что привязано, что нет, есть ли конфликты.
Показать сводку «что нашли» и предложить действия:
- ПОДХВАТИТЬ существующие УТМ (attach: записать в state.json, дальше рулить ими);
- настроить с нуля (мастер установки);
- ничего не делать (только мониторинг).
Экран «первого запуска» в вебе; чтение токенов делегируется трею (см. связку веб↔трей).

## Централизованное управление парком (ПОСЛЕ доработки локального оркестратора)
Центральный пульт, который видит и управляет многими машинами:
- машины с нашим оркестратором (несколько УТМ на токенах);
- машины со ШТАТНЫМ одиночным УТМ (без оркестратора).
Ключевые куски:
1. Режим «одиночный УТМ» у оркестратора: ставим оркестратор на машину с одним
   обычным УТМ — он просто мониторит/рулит им (без многотокенной хореографии;
   один токен садится штатно). Т.е. оркестратор как универсальный агент.
2. Оркестратор отдаёт API по СЕТИ (не только localhost) — с авторизацией
   (логин/пароль «по галочке» — каркас есть) + IP-allowlist (каркас есть).
3. Центральный хаб (веб-пульт): регистрирует оркестраторы по адресам, агрегирует
   статус всего парка, шлёт команды (перезапуск/обновление/лечение), алерты.
4. Безопасность канала (TLS), учёт машин, роли.
Очерёдность: сперва довести локальный оркестратор (папка, установка, web↔трей,
проверки), потом браться за централизацию.

## Перенос УТМ с компьютера на компьютер
УТМ = токен (Rutoken/ФСРАР-ИД — это и есть личность) + папка transporter (transportDB
+ conf + ключи) + регистрация службы. Перенос A→B:
- физически переносим ТОКЕН (Rutoken) с A на B — личность переезжает с ним;
- вариант «чисто»: на B ставим свежий УТМ на этот токен (новый transportDB) — просто,
  но теряется история/незавершённые обмены;
- вариант «полный»: экспорт папки УТМ (transportDB+conf) с A → импорт на B →
  регистрация службы → привязка токена (introduce) — сохраняет состояние;
- на A: снять/удалить УТМ (стоп службы, убрать папку), чтобы не задвоился.
Оркестратор автоматизирует: «Экспорт УТМ» на A (упаковать папку + порт/ФСРАР),
«Импорт УТМ» на B (распаковать + зарегистрировать + привязать). Токен переносится
руками. transportDB копируется файлом (УТМ на A остановить, чтобы не был занят).
Через связку веб↔трей (регистрация службы и привязка — session-0/introduce; чтение
токена — трей). Учесть: один ФСРАР не должен работать на двух машинах одновременно.
