# utm-orchestrator — беклог и архитектура

Живой документ. Составлен 2026-07-10 на основе всего, что проверено на боевой
машине (см. `NOTES.md` — там доказательства и механика). Реализуем поэтапно.

## 0. Главные принципы (не нарушать)

- **Не ломать работающее.** Любая операция с ридерами меняет ГЛОБАЛЬНУЮ
  PC/SC-таблицу и может сломать другие УТМ. Здоровые УТМ не трогаем никогда;
  чиним только сбойные и только точечно.
- **Идемпотентность и безопасный повтор.** Любой шаг можно прервать/повторить
  без порчи состояния. Перед действием — читаем реальное состояние, потом
  решаем.
- **Отказоустойчивость по частям.** Если один/несколько УТМ не поднимаются —
  остальные работают. Сбойные помечаем: статус + причина (нет токена, нет
  папки, порт занят, БД повреждена, и т.д.).
- **Серийник токена — первичный ключ** (PKCS11 `C_GetTokenInfo.serialNumber`,
  hex). ФСРАР-ИД — вторичный (может отсутствовать у пустого токена, появляется
  после выпуска подписи). Device-имя ридера НЕ использовать для идентификации
  (самореференциально, хрупко).
- **Всё логируется**, пошагово, по каждому УТМ, с ротацией.
- **Разработка на боевой машине — только безопасно.** Пишущие/деструктивные
  операции тестируем в контролируемом окне или на отдельных портах/папках; по
  умолчанию — read-only.

## 1. Поддерживаемые состояния машины (при установке/подхвате)

- **Чистая**: нет УТМ, нет 2UTM, нет нас.
- **Есть один УТМ** (вендорская установка) — подхватить.
- **Есть 2UTM** — сосуществовать/мигрировать (снять его автозапуск, перенять
  управление; НЕ ломать, пока не подтверждено).
- **Есть мы** (наша прошлая установка) — обновить/переконфигурировать.

## 2. Поддерживаемые состояния токенов

- Один токен / много токенов.
- «Чистый» токен (без сертификатов).
- Токен без ФСРАР-ИД, но с одним/несколькими другими сертификатами.
- Токен с ФСРАР-ИД (готовый).
- Токен физически «уснул» (ридер есть, карта не отвечает) → лечится
  программным сбросом (forget + SCardSvr restart) или переподключением.

## 3. Ключевая механика (проверено, см. NOTES)

- **RSA-путь при ЗАГРУЗКЕ читает строго слот 0.** В рантайме переиниц. НЕ
  читает слот 0 (держит сессию/ищет по идентичности). Значит: каждый УТМ должен
  стартовать, когда ЕГО токен на слоте 0 → службы поднимаем ПО ОДНОЙ на каждой
  загрузке ОС (не Windows auto-start скопом).
- **Peel-down bring-up (надёжный, без хрупких rename):**
  1. Программный сброс → детерминированный нативный порядок.
  2. Читаем FSRAR/serial на слоте 0 → находим его службу → запускаем → ждём
     готовности (служба «схватывает» токен).
  3. `SCardForgetReader` ридера слота 0 → на слот 0 встаёт следующий токен.
  4. Повтор, пока не поднимутся все.
  5. Программный сброс (вернуть все ридеры для рантайма).
- **Установка «с нуля» без вендорского setup.exe**: копируем распакованный
  источник (innoextract) → папки, правим порты + PIN, procrun-служба,
  choreography. Проверено на 6 токенах.
- **Внутренние порты** (схема 2UTM): web = задан; transport = 8192 − i;
  agent = 8194 + i (i — номер экземпляра; UTM_1/base = 8080/8192/8194).

## 4. Компоненты

- **core** — библиотека: сканер токенов (serial+fsrar+slot+reader), ReaderTable
  (winscard: list/attr/forget/introduce + программный reset), ServiceControl
  (procrun install/delete, sc start/stop, ожидание готовности по HTTP),
  HTTP-диагностика (`/api/info/list` → ownerId/rsaError), лог.
- **state** — единый файл состояния (serial→port→service→fsrar→version, ссылка
  на PIN, статусы/причины сбоев). + блокировка параллельного доступа.
- **orchestrator-service** — Windows-служба: boot-bring-up, периодический
  health-check + точечное восстановление, слежение за версией УТМ и за
  релизами нашего оркестратора на GitHub.
- **panel** — веб-интерфейс (локально + удалённо/телефон), аутентификация,
  мониторинг, действия (install/update/remove/add/attach, порты, firewall,
  логи).
- **installer/updater** — установка/подхват/удаление УТМ, обновление УТМ с
  сохранением transportDB/conf/l (логика Update-UTM.ps1), самообновление
  оркестратора.

## 5. Поэтапный беклог

### P0 — Фундамент (безопасно, без изменений на машине)
- [ ] Решить стек (см. §6 «Открытые решения»).
- [ ] Структура репо, сборка, CI.
- [ ] `core`: сканер токенов по **серийнику** (+fsrar/slot/reader), ReaderTable
      с программным reset, ServiceControl, HTTP-диагностика, структурный лог.
- [ ] Модель `state` (файл состояния + схема) и блокировка.
- [ ] Всё в read-only режиме проверяемо на живой машине (только чтение).

### P1 — Boot-оркестрация (СРОЧНО: чинит ребут)
- [ ] Перевести службы на **manual start**.
- [ ] **Peel-down bring-up** как процедура + запуск при старте Windows
      (Windows-служба или задача планировщика с наивысшими правами).
- [ ] Пер-УТМ статус: OK / faulty+причина. Один сбойный не блокирует остальные.
- [ ] Подробный лог bring-up.
- [ ] Безопасный тест: в контролируемом окне (например, когда пользователь и так
      готов к ребуту).

### P2 — Периодический health-check + точечное восстановление
- [ ] Раз в ~1.5 часа: **read-only** проверка (rsaError, наличие токенов,
      папок, портов).
- [ ] Всё ОК → ничего не трогаем.
- [ ] Токен пропал → пометить УТМ faulty + причина; при возврате токена —
      **точечно** поднять только его (мини-peel-down), не трогая здоровые.
- [ ] УТМ сломан, но токен на месте → точечно перезапустить/перепривязать.
- [ ] Алертинг (в панель; позже — push/почта).

### P3 — Установка / жизненный цикл УТМ
- [ ] Определение состояния машины (чистая / один УТМ / 2UTM / мы).
- [ ] Определение состояния токенов (с/без ФСРАР, с иными сертификатами).
- [ ] Скачивание дистрибутива УТМ + распаковка (innoextract) как источник.
- [ ] install / add / remove / **attach existing** (подхватить имеющиеся).
- [ ] Управление портами.
- [ ] Правила файрвола Windows (открыть/закрыть порты УТМ).
- [ ] Хранение PIN (безопасно).

### P4 — Панель управления + мониторинг (веб)
- [ ] Локальный веб-сервер, аутентификация (пароль + TLS).
- [ ] Дашборд: статус каждого УТМ (порт, ФСРАР, токен, обмен, ошибки).
- [ ] Действия: install/update/remove/add/attach, порты, firewall.
- [ ] Просмотр логов по каждому УТМ.
- [ ] Удалённый доступ + мобильный вид.
- [ ] Упаковка в единый exe.

### P5 — Обновления
- [ ] Обновление УТМ с сохранением transportDB/conf/l.
- [ ] Раз в неделю проверка новой версии УТМ → показать «доступно обновление».
- [ ] Слежение за релизами оркестратора на GitHub → предложить самообновление.

### P6 — Резервное копирование (версия 3, не раньше)

- [ ] Удалённый бэкап: сетевые хранилища, S3.

## 6. Открытые решения (обсудить до P0)

- **Стек — РЕШЕНО: C#/.NET** (2026-07-10). Нативная интеграция с Windows
  (служба, файрвол, реестр, procrun), одиночный self-contained exe, встроенный
  веб (Kestrel/minimal API), P/Invoke к winscard (сигнатуры у нас есть из
  reader-binder). Существующие Java-инструменты (reader-binder/TokenScanner) —
  как справочная реализация механики и для промежуточного boot-fix, пока
  портируем логику в C#.
- **Где живёт boot-оркестрация**: Windows-служба vs задача планировщика (нужны
  наивысшие права — вчера выяснили, что elevation критичен).
- **Хранение PIN**: DPAPI (Windows) / зашифрованный файл.
- **Безопасность панели**: локальный пароль + TLS; удалённый доступ — через
  что (проброс порта / VPN / туннель).
- **Тест-стратегия на боевой машине**: read-only по умолчанию; деструктив —
  только в согласованном окне.
