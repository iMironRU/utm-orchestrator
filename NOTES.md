# Заметки по проекту utm-binding-agent / utm-orchestrator

Дата: 2026-07-08. Живая отладочная сессия на боевой машине с 6 экземплярами
UTM ЕГАИС и 6 токенами Rutoken ECP. Здесь — всё, что узнали, чтобы не терять
между сессиями.

## 1. Исходная проблема

2UTM (github.com/2UTM/2UTM) позволяет держать до 10 УТМ на одной машине, но
привязка токена к конкретному экземпляру UTM — **позиционная**: если токены
переставить местами (или производится обслуживание), UTM может подхватить
"не свой" сертификат. Наша цель — привязывать по **FSRAR-ID сертификата**,
а не по физической позиции.

## 2. Механизм самого UTM (ru.centerinform.transport, версия 4.2.0)

- Java 8, Spring Boot, Tomcat embedded, порт из `transport.properties`
  (`web.server.port`).
- PKCS#11 доступ идёт через **ДВЕ РАЗНЫЕ** библиотеки-обёртки одного и того
  же драйвера Rutoken:
  - `gost.pkcs11.library.path=C:\Windows\SysWOW64\rtPKCS11ECP.dll` — для
    ГОСТ-крипто (реальный обмен документами с ЕГАИС).
  - `rsa.library.path=C:\Windows\SysWOW64\rtPKCS11ECP-replica.dll` — для
    RSA (используется в проверке FSRAR_ID при инициализации/переинициализации).
- Выбор слота для **GOST**-пути идёт через
  `ru.centerinform.transport.crypto.SunCryptographer.c(String requestedModel)`
  (возвращает `long` — номер слота). Именно этот метод патчит наш агент.
- Выбор слота для **RSA**-пути идёт **ДРУГИМ, ещё не найденным путём** —
  скорее всего где-то в обфусцированном `ru.centerinform.transport.pki.key.KeyMaster`
  (jar `terminal-pki-4.2.0.jar`). Кандидат — приватный метод `v()` (возвращает
  `long`), но не подтверждено. **Наш Java-агент чинит только GOST-путь.**
- `KeyMaster.r()` (KeyMaster.java:343) выбрасывает
  `IllegalStateException: FSRAR_ID (CN RSA сертификата) на используемом ключе
  не соответствует FSRAR_ID первичной инициализации УТМ.`, если CN RSA-сертификата
  не совпадает с `db.ownerId`.
- **`db.ownerId`** — поле в `transportDB` (встроенная Derby, зашифрована,
  `dataEncryption=true` в `service.properties`), которое **записывается
  один раз, автоматически, при первом успешном чтении RSA-сертификата после
  создания свежей БД**, и потом уже не меняется штатными средствами. Смотрится
  через `GET /api/info/list` → `db.ownerId` / `db.createDate`.
  Мы наступили на грабли: при переустановке UTM "с нуля" transportDB
  создалась и сразу подцепила **не тот** сертификат (позиционно, слот 0),
  зафиксировав это значение навсегда для этого экземпляра — из-за этого
  RSA-сравнение "случайно" не падало (обе стороны сравнения были одинаково
  неверны), хотя реальная привязка была неверной.

## 3. Полезные HTTP-эндпоинты UTM (localhost:<port>)

- `GET /diagnosis` — XML с текущим сертификатом (`<CN>`, `<OU>` и т.д.).
  **НЕНАДЁЖНО**: похоже, читает RSA-путь либо кэш, не всегда актуально.
- `GET /api/rsa/orginfo`, `GET /api/gost/orginfo` — JSON с cn/serial/датами
  сертификата каждого пути отдельно. `gost/orginfo` корректно отличается
  между экземплярами (наш фикс работает), `rsa/orginfo` — нет (проблема RSA-пути).
- `GET /api/info/list` — **самый информативный**: содержит `rsaError`,
  `ownerId`, `db.ownerId`, `db.createDate`, `rsa{...}`, `gost{...}`. Именно
  этот эндпоинт использует сам 2UTM для проверки (`rsaError=null` → всё ок).
- **Самый надёжный сигнал в принципе** — реальный `transport_info.log`:
  ищи `не соответствует FSRAR_ID первичной инициализации` (провал) или
  `Завершение задачи обмена документами с сервером ЕГАИС по расписанию`
  без ошибок перед этим (успех). `WSExchangeJob` гоняется по расписанию
  (не только на старте).

## 4. Наш агент (utm-binding-agent) — как читает FSRAR-ID без PIN

- Изначально пытались через reflection в существующий
  `sun.security.pkcs11.wrapper.PKCS11` (JDK-обёртка) — **не работает**:
  JDK-обёртка сама решает, когда идти по двухфазному протоколу
  `C_GetAttributeValue` (запрос длины → аллокация → повторный запрос), и для
  этого конкретного драйвера Rutoken это не срабатывает (pValue тихо
  остаётся null, без исключения, независимо от логина).
- Рабочее решение: **`ru.rutoken:pkcs11jna:4.3.0`** (JNA-обёртка от Aktiv) —
  прямой вызов `rtPKCS11ECP.dll`, свой независимый от UTM PKCS#11-контекст,
  вручную реализованный двухфазный протокол (см. `SlotFilter.readAttr()`).
  Не требует C_Login — сертификаты читаются анонимно (проверено эмпирически:
  CKA_CLASS, CKA_SUBJECT, CKA_VALUE — всё доступно без логина при правильном
  протоколе).
- FSRAR-ID лежит в **CN** субъекта сертификата (не в отдельном OID, как
  предполагалось изначально) — формат `03XXXXXXXXXX`, 12 цифр. У личного
  сертификата сотрудника (второй сертификат на той же токене) CN — ФИО,
  поэтому ищем среди всех найденных сертификатов тот, где CN похож на
  FSRAR-ID (`\d{10,14}`).
- `C_Initialize` можно смело вызывать даже если UTM уже сделал это в своём
  процессе — драйвер терпит повторную инициализацию (не бросает
  `CKR_CRYPTOKI_ALREADY_INITIALIZED` в наших тестах, но код на это готов).
  **НИКОГДА не вызывать `C_Finalize`** — это убьёт открытую сессию UTM в этом
  же процессе.

## 5. 2UTM — как это делает "правильный" инструмент (github.com/2UTM/2UTM)

Исходники: C++, `src/ControlReaders.cpp`, `src/config.cpp`.

- Работает через **Windows WinSCard API** — `SCardListReaders`,
  `SCardGetAttrib(..., SCARD_ATTR_DEVICE_SYSTEM_NAME, ...)`,
  `SCardForgetReader`, `SCardIntroduceReader`.
- Хранит соответствие в `config.ini` (рядом с exe, обычно
  `C:\Program Files (x86)\2UTM_vs\config.ini`):
  ```ini
  [UTM_1]
  nameReader=Aktiv Rutoken ECP 1
  attrReader=Aktiv Rutoken ECP 1
  serialNumber=1157188378   ; ДЕСЯТИЧНЫЙ! перевести в hex для сверки с нашими данными
  port=8080
  ```
- **ВАЖНО**: `attrReader` в текущих наблюдениях всегда равен `nameReader` —
  `SCARD_ATTR_DEVICE_SYSTEM_NAME` возвращает **текущее** имя ридера
  (самореференциально), не какой-то независимый серийник железа.
- FAQ самого 2UTM: "ридеры пропадают" — чинится только через физическое
  отключение/переподключение всех смарт-карт (диспетчер устройств →
  отключить все → подключить все). **Мы столкнулись с этим сами** (см. ниже).
- Служебные порты (8192, 8194) 2UTM тоже двигает по инстансам (±1 на экземпляр).

## 6. Эксперимент со свопом ридеров — что подтвердили и что не удалось

- **Подтверждено**: RSA-путь (`rtPKCS11ECP-replica.dll`) действительно
  **позиционный** — после свопа `SCardForgetReader`+`SCardIntroduceReader`
  сертификат на RSA-пути реально сменился на обоих портах одновременно
  (обе службы получили рассогласование, которого не было до свопа — то есть
  до свопа совпадение было случайным из-за одинаково неверного `db.ownerId`,
  см. п.2).
- **Не удалось**: обратный своп (тот же скрипт повторно) **не вернул**
  исходное состояние — `SCardForgetReader`/`SCardIntroduceReader` оказались
  менее детерминированными, чем ожидалось (подозрение: Windows PnP
  переопределяет что-то после Forget по своей логике, независимо от
  следующего Introduce).
- **Восстановились** только через физическое отключение/переподключение
  токенов (тот самый пункт FAQ 2UTM) — после этого `slot0`/`slot1` вернулись
  к исходному соответствию, и после перезапуска служб всё снова заработало
  штатно (реальный обмен документами, без ошибок).
- Скрипты эксперимента (черновики, не финальная реализация):
  `Get-ReaderAttrs.ps1`, `Swap-Readers.ps1` — лежат в scratchpad сессии,
  **не в репозитории**. При реализации `reader-binder` компонента (см. TODO)
  нужно делать надёжнее: проверять состояние до и после каждой операции,
  иметь понятный откат/восстановление, не полагаться на "симметричность"
  Forget+Introduce.

## 6.5. Как 2UTM реально устанавливает привязку — механизм (разобрано 2026-07-09)

Полный разбор `installUTM()` в `2UTM_vs.cpp` (не только `ControlReaders.cpp`).
Ключевой вывод: **никакой статичной привязки "слот N ↔ токен" не существует** —
ни на уровне PKCS11, ни через `config.ini` (там просто **финальное желаемое
имя ридера**, а не что-то, что само по себе гарантирует нужный слот). Вся
магия — в **порядке и таймингах**:

1. `readConfigNameAndReader()` читает `config.ini` в порядке `i = countUTM..1`
   (т.е. от `UTM_N` до `UTM_1`).
2. Забыть **все** текущие ридеры (`SCardForgetReader` на каждый).
3. Цикл `i = 0..N-1`: на каждой итерации — **ввести РОВНО ОДИН** ридер
   (`SCardIntroduceReader`) с именем `"Aktiv Rutoken ECP " + (N-1-i)`
   (т.е. номера убывают: сначала самый старший, в конце — `... 0`), **сразу
   запустить СООТВЕТСТВУЮЩУЮ службу**, и **дождаться** её готовности
   (`checkHomePageUTM` — опрос HTTP) **прежде чем** переходить к следующей
   (более низкой) итерации.
4. Порядок запуска служб — обратный номерам: сначала `TransportN` (самый
   старший), в конце — голый `Transport` (UTM_1).

Почему это работает, хотя сам UTM (что GOST-, что RSA-путь) при каждом
`SunCryptographer.c()`/аналоге **слепо берёт "слот 0"**, без всякой
конфигурации per-instance: каждый JVM делает этот выбор **один раз**, при
первом обращении к PKCS#11 в своём процессе. В момент старта `TransportN`
существует **только его собственный** ридер → он тривиально единственный
и является "слотом 0" → блайнд-выбор совпадает с правильным. Далее вводится
ридер с **более низким** номером для следующей службы — PKCS11 сортирует
слоты по имени ридера, поэтому новый, более низкий номер становится новым
"слотом 0" **до** того, как следующая служба успевает сделать свой
единственный, one-time выбор. Повторить до `"...ECP 0"` и голого `Transport`.

**Важно**: после того как эта "пляска" выполнена один раз, привязка имени
ридера к физическому устройству (`SCardIntroduceReader`) **сохраняется между
перезагрузками ОС**, если токены остаются в тех же физических USB-портах —
именно поэтому 2UTM решает исходную проблему навсегда, а не только "здесь
и сейчас", хотя сам UTM никогда не перестаёт слепо брать "слот 0". Но
переподключение токена **в другой** физический порт (или, как выяснилось на
практике, иногда даже в тот же самый) может сбросить это состояние — тогда
нужно повторить всю процедуру заново.

## 6.6. `reader-binder` — наша реализация той же идеи, но по FSRAR-ID

Новый модуль `reader-binder/` (Maven, JNA): переносит принцип 2UTM на нашу
уже проверенную схему идентификации токена (FSRAR-ID из CN сертификата,
без PIN, тот же код что в `agent/SlotFilter.java`), вместо серийника из
`config.ini`. Ключевые классы:

- `TokenScanner` — сканирует PKCS11-слоты, для каждого возвращает
  `CK_SLOT_INFO.slotDescription` (имя ридера) + FSRAR-ID сертификата.
- `ReaderTable` — обёртка над `winscard.dll` (`SCardListReaders`,
  `SCardGetAttrib(SCARD_ATTR_DEVICE_SYSTEM_NAME)`, `SCardForgetReader`,
  `SCardIntroduceReader`). **Важная деталь реализации**: `SCARDCONTEXT`/
  `SCARDHANDLE` в WinAPI — это `ULONG_PTR` (размер указателя, 8 байт на x64),
  а не `long` (4 байта всегда на Windows) — в JNA нужно использовать
  `Pointer`/`PointerByReference`, а не `NativeLong`, иначе будет переполнение
  буфера при заполнении out-параметров.
- `BindingConfig` — грузит целевую привязку `fsrarId → (индекс, служба, порт)`
  из простого `bindings.properties` (в репозитории, с комментариями по каждой
  ещё не подтверждённой записи).
- `ServiceControl` — `sc.exe start/stop` + опрос HTTP до готовности. **Грабли**:
  первая версия ждала фиксированные 500мс между stop и start — `sc.exe`
  возвращает управление, как только запрос принят, а не когда служба реально
  дошла до `STOPPED`; из-за этого `start` иногда получал ошибку 1056 ("уже
  запущена"), хотя служба на самом деле была ещё `STOP_PENDING`, и в итоге
  обе службы просто падали в `STOPPED` без реального рестарта. Исправлено:
  `ServiceControl` теперь **реально опрашивает** `sc query` до нужного
  состояния, а не спит фиксированное время.
- Режимы: `--verify` (только чтение, безопасно), `--dry-run` (план без
  выполнения), `--apply` (реально выполняет: forget всех → по каждой записи,
  от старшего индекса к младшему — introduce → restart службы → ждать HTTP).

### Инцидент 2026-07-09 при первом `--apply` — разбор

Первый прогон `--apply` (до исправления гонки stop/start) уронил обе службы
в `STOPPED` без перезапуска. После ручного `sc.exe start` (почти
одновременно для обеих) у **обеих** служб RSA-путь стал не просто путать
сертификат, а **падать с ошибкой** (`KeyStore is not initialized`,
`db.ownerId` в `/api/info/list` стал `null` вместо реального значения) —
реальный обмен документами отменялся (`Задача обмена документами отменена.
Ошибка работы RSA`).

Проверили и **отвергли** две версии:
- ~~Агент подменяет слот и для RSA-пути тоже~~ — нет, в логе агента только
  ОДИН вызов `SunCryptographer.c()` на перезапуск (агент трогает только
  ГОСТ-путь, как и задумано).
- ~~Служба `SCardSvr` осталась в плохом состоянии~~ — её перезапуск (после
  предварительной остановки Transport/Transport2, т.к. `sc stop SCardSvr`
  сам не может остановить службу с зависимыми — ошибка 1051) не помог сам
  по себе.

**Реальная причина** оказалась банальнее и важнее: `IllegalStateException:
KeyStore is not initialized` — это безобидная гонка при **любом** старте
(запланированная `WSExchangeJob` иногда успевает выполниться раньше, чем
`KeyMaster` заканчивает собственную инициализацию RSA) — она не новая и не
из-за наших действий. А вот сам `rsaError` (`FSRAR_ID ... не соответствует`)
для `Transport` — это **тот самый, изначально описанный в §2 баг**
(`db.ownerId` навсегда испорчен при переустановке), который раньше не был
виден только потому, что "родной" (позиционный) слот 0 **случайно** совпадал
с этим же неверным значением. Как только физическое переподключение токенов
поменяло физический порядок местами, слот 0 стал **честно** показывать
правильный сертификат Transport — и **впервые по-настоящему** проявилось
несовпадение с испорченным `db.ownerId`. То есть: **это не регрессия от
reader-binder** — это тот же старый баг, который стал видимым.

Отсюда следует важная ранее не проверенная деталь: **RSA-рассогласование
реально ОТМЕНЯЕТ обмен документами**, а не является чисто
диагностическим/некритичным — то, что Transport "работал" раньше, было
чистой удачей совпадения, а не свидетельством, что ГОСТ-пути достаточно.

**Итог после исправления гонки stop/start и повторного (уже согласованного)
`--apply`**:
- **Transport2 (8081): реально исправлено.** `rsaError: null`, `db.ownerId`
  совпадает (`030000903354`=`030000903354`), RSA и ГОСТ оба валидны,
  подтверждено свежим `Завершение задачи обмена документами...` в логе.
- **Transport (8080): ГОСТ по-прежнему верный (агент), RSA — по-прежнему
  ошибка**, потому что `db.ownerId` навсегда испорчен (см. §2) — reader-binder
  не может это починить сам по себе, живое чтение (правильное, `353`) никогда
  не совпадёт с испорченным сохранённым значением (`354`). Нужен отдельный
  шаг: сброс/пересоздание `transportDB` для этого экземпляра, выполненный
  ПОСЛЕ того, как reader-binder уже корректно закрепил его собственный
  токен — тогда первое чтение после пересоздания БД зафиксирует правильный
  `ownerId`. Это отдельное, более рискованное действие (удаление БД) —
  требует отдельного явного согласия перед выполнением.

## 6.7. Попытка починить Transport через сброс БД и переустановку (2026-07-09, продолжение)

Пробовали довести Transport до конца, по инструкции пользователя "чиним
транспорт потом все остальные". Три отдельных попытки, каждая давала новую
информацию:

**Попытка А — просто пересоздать `transportDB`.** Остановили Transport,
перенесли (не удалили) `transportDB` в бэкап, дали стартовать заново.
UTM отказался создавать новую БД: `IllegalStateException: Отсутствует
пароль RSA KeyStore: config/transport.properties ->

## 6.8-6.9. РЕШЕНО: RSA-путь выбирает сертификат по идентичности, не по позиции

Изначально (см. старую версию этого раздела в истории git) казалось, что
`reader-binder` не может стабильно управлять RSA-путём (`KeyMaster` /
`rtPKCS11ECP-replica.dll`) — Transport после подтверждённой правильной
привязки ридеров всё равно читал сертификат `354` (Transport2). Проверили
экспериментом (см. §6.7 «Попытка А», и продолжение ниже) и **выяснили
настоящую причину — она гораздо проще и важнее**:

**Эксперимент**: поменяли местами индексы в `bindings.properties` (Transport
→ индекс 0 = "слот 0", Transport2 → индекс 1), прогнали `reader-binder --apply`
заново. Результат:
- **Transport**: `KeyMaster` использовал сертификат `SerialNumber:
  198332db00000060395f` — **абсолютно тот же самый serial**, что и в ДВУХ
  предыдущих перезапусках, ДО этого эксперимента, когда физический порядок
  слотов был другим. Слот сменился, порядок ридеров сменился — а выбранный
  сертификат остался ТЕМ ЖЕ. Это прямое опровержение позиционной гипотезы:
  если бы `KeyMaster` слепо брал "слот 0", после смены порядка он должен был
  бы взять ДРУГОЙ физический сертификат.
- Вывод: **`KeyMaster` ищет RSA-сертификат активным поиском по совпадению
  CN с сохранённым `db.ownerId`** — точно так же, как наш собственный агент
  ищет ГОСТ-сертификат по FSRAR-ID (см. §4). Позиция/имя ридера тут вообще
  не участвует в выборе. Именно поэтому `reader-binder` никогда не мог и не
  должен был влиять на то, какой сертификат читает RSA-путь — это была не
  та рукоятка управления с самого начала.
- Из этого следует прямо и однозначно: **единственная причина, по которой
  RSA-путь Transport не совпадает — это испорченный `db.ownerId=354`**
  (см. §2, изначальный диагноз в самом начале сессии). Никакая работа с
  ридерами это не чинит и не может починить. Единственный настоящий фикс —
  сбросить/пересоздать `transportDB` для Transport, чтобы первое (уже верно
  читаемое, `353`) значение записалось как новый `db.ownerId`.

**Побочный эффект эксперимента**: пока PC/SC-таблица была неполной
(Transport2 стартовал, когда был введён только его собственный ридер, до
того как ридер Transport вернули обратно), Transport2 поймал `IllegalStateException:
... не соответствует ...` при самой первой попытке (не мисматч по значению,
а гонка — в этот момент `KeyMaster` смог обнаружить только 1 слот и не смог
корректно его прочитать), и потом продолжал видеть только 1 слот и падать
при каждой периодической переинициализации (`WSExchangeJob` каждые ~2 мин),
**не восстанавливаясь сам**. Откатили `bindings.properties` к исходному
порядку и просто **перезапустили Transport2 отдельно** (без нового
forget/introduce — имена ридеров и так были уже правильные по `--verify`) —
это полностью починило зависшее состояние процесса. Практический вывод:
если после `reader-binder --apply` какая-то служба "залипла" на неполном
представлении слотов — **простой перезапуск именно этой службы** (без
повторного трогания ридеров) обычно достаточен, раз имена ридеров уже верны.

**Итог**: Transport и Transport2 оба вернулись в исходное рабочее состояние
(реальный обмен документами подтверждён у обоих в логах). `bindings.properties`
восстановлен к исходному порядку (Transport2→0, Transport→1) — порядок
индексов здесь теперь имеет смысл ТОЛЬКО как ground truth для физической
идентификации токенов (через `--verify`), а не как механизм влияния на
RSA-путь.

## 6.10. Попытка №2 пересоздать transportDB для Transport — новые находки

Продолжили попытки почитить Transport (см. §6.7 попытка А) с учётом вывода
§6.9. Пользователь подтвердил (сославшись на исходники 2UTM — там ДЕЙСТВИТЕЛЬНО
нет ни единого упоминания `transportDB`/`keystorePassword`/`PIN`, см. ниже),
что 2UTM никогда сам не создаёт БД с нуля — значит, наша ситуация (создание
с нуля) генуинно отличается от штатного сценария 2UTM.

**`crypto.lib.pki.keystorePassword` и `crypto.lib.gost.keystorePassword` —
это НЕ пароли локальной БД, это НАСТОЯЩИЕ PIN-коды смарт-карты.** Пусты они
в РАБОЧЕМ `transport.properties` потому, что **UTM сам стирает значение
сразу после успешного использования** (защита от хранения PIN в открытом
виде на диске). Подтверждено экспериментально:
- Пустое поле → `IllegalStateException: Отсутствует пароль RSA/GOST KeyStore`
  при попытке создать НОВУЮ БД (при открытии уже существующей — не
  проверяется).
- Произвольное (случайное) значение → проходит проверку "непусто", но
  падает позже с `CKR_PIN_INCORRECT` при реальном `C_Login` в
  `SunCryptographer`/`P11KeyStore` — то есть это ПРОВЕРЯЕТСЯ токеном,
  не просто локальный пароль.
- Правильный PIN (в этой организации — стандартный заводской PIN, одинаковый
  на всех токенах; см. пароль-менеджер/private-заметки, не хранить здесь
  открытым текстом) → создание БД успешно проходит до конца.
- **Оба поля (`pki` и `gost`) нужно проставлять ОДНОВРЕМЕННО** перед
  запуском — если хотя бы одно проверяется раньше и падает, то УЖЕ
  проверенное успешно поле к этому моменту уже стёрто, и следующий
  перезапуск начинает сначала с "поле пусто" по НЕМУ.

**НО даже после успешного создания БД с правильным PIN, `db.ownerId`
снова записался как `354` (чужой), а НЕ `353`.** Проверили дважды:
1. Без переименования ридеров (естественный порядок слотов на тот момент).
2. С переименованием через `reader-binder` ПРЯМО ПЕРЕД пересозданием БД
   (Transport гарантированно на "слоте 0" по данным `--verify`).

**Оба раза результат одинаковый — `354`.** Значит, ни естественный
порядок, ни намеренно проставленный через reader-binder порядок слотов НЕ
влияют на то, что записывается при первой инициализации. Гипотеза "первая
инициализация слепо берёт слот 0" (выдвинутая как объяснение в первой
версии этого раздела) **тоже, получается, неверна** — что-то ещё
определяет, какой сертификат становится "первичным", и это что-то мы
пока не нашли чисто программным путём.

**Важнейшая подсказка от пользователя** (по памяти о том, как изначально
настраивался этот же самый набор из 6 токенов): "я вставил токены запустил
2УТМ он их увидел и попросил инсталляционный файл УТМ потом всё установил
на нужные порты... я зашёл через браузер в каждый утм по порядку и выпустил
ФСРАР подписи через интерфейс УТМ — они записались каждый на тот токен,
который связан с конкретным утм". То есть **`db.ownerId` не определяется
пассивно первым чтением при старте JVM — есть штатное действие через
веб-интерфейс UTM ("выпустить ФСРАР подпись"), которое явно регистрирует
привязку**, и именно это действие пользователь выполнял вручную, one at
a time, для каждого экземпляра при первоначальной настройке и при
предыдущих поломках ("так уже было, я лечил как удалял всё и заново
устанавливал, потом смотрел в каком утм какой ФСРАР ИД видится").

**Это меняет приоритет следующего шага**: вместо того чтобы продолжать
бороться с пассивным определением при старте, нужно **найти и
воспользоваться этим разделом веб-интерфейса** (вероятно что-то вроде
"регистрация"/"выпуск сертификата"/"первичная настройка" в UI на
`http://localhost:8080`). Не удалось исследовать в этой сессии — расширение
Chrome для браузерных инструментов было недоступно (`Chrome extension isn't
connected`), а `computer-use` даёт только просмотр экрана без кликов для
браузеров. **TODO для следующей сессии**: подключить браузерные инструменты
и найти эту функцию через UI напрямую.

**Побочная находка — нестабильность агента после множества циклов
stop/start без переустановки службы.** В какой-то момент за сегодня наш
javaagent перестал подгружаться на перезапуск службы (ни одного нового
файла лога), несмотря на корректный параметр `-javaagent:...=fsrarId=...`
в реестре Procrun (`HKLM\...\Procrun 2.0\Transport\Parameters\Java\Options`).
Причина не установлена (подозрение на какой-то кеш на уровне JVM/procrun
после десятков рестартов подряд) — эта нестабильность прошла сама после
полной перезагрузки ОС, которую сделали по этой же причине.
**Итог сессии на момент перезагрузки**: обе службы вернулись к ИСХОДНОМУ
"случайно работающему" состоянию (Transport2 — по-настоящему верно;
Transport — ГОСТ верно через агент, RSA случайно совпадает с испорченным
`db.ownerId=354`) — то есть по факту db.ownerId для Transport ПО-ПРЕЖНЕМУ
не исправлен, несмотря на несколько успешных пересозданий БД с правильным
PIN — потому что первичная привязка каждый раз записывает `354`, а не `353`,
неизвестным нам пока способом.

## 7. Технические грабли по пути (для истории, если решим чинить Java RSA-путь)

- **Классическая ошибка**: `IllegalAccessError` при вызове
  `appendToBootstrapClassLoaderSearch()` в старой версии агента — было нужно
  для инструментирования bootstrap-класса `PKCS11` (первый, отброшенный
  подход), но лишнее и вредное для текущего подхода (инструментируем
  `SunCryptographer`, который грузится обычным app classloader'ом). Убрано.
- **Реальные сигнатуры JDK PKCS11-обёртки** (эта конкретная сборка,
  проверено через `javap` на `sunpkcs11.jar` из `C:\UTM\jre\lib\ext\`):
  - `C_OpenSession(long, long, Object, CK_NOTIFY)` — не `(long,long,Object,Object)`.
  - `C_FindObjectsInit(long, CK_ATTRIBUTE[])`, `C_GetAttributeValue(long, long, CK_ATTRIBUTE[])`
    — не `Object[]`.
  - `C_FindObjects(long, long)` — **`long`, не `int`** для maxObjectCount
    (нетипично, отличается от некоторых других сборок JDK).
  - Использовать `javap` на реальном `sunpkcs11.jar` целевой машины, не
    полагаться на память/доки — сигнатуры реально отличаются между сборками.
- Обфускация `terminal-*.jar`: строки шифруются XOR посимвольно + вызов
  дешифратора вида `ru/centerinform/transport/pki/key/b$b.E(Object, short, short, short)`
  (или похожий, имя класса/метода может отличаться между jar). Полная
  расшифровка не делалась — слишком трудозатратно для разового анализа.

- `agent/` — Java-агент (Maven, JDK 8 target). Ключевые классы:
